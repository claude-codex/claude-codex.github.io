Section 0

00TL;DR — what the paper contributes

A source-grounded design-space analysis. Not a benchmark paper. The contribution is a vocabulary.

Three things I'd cite this forCitable
  1. The 1.6% / 98.4% split. Quantitative spine of the argument that the harness, not the model, is now the differentiator as frontier coding capability converges. Positions Claude Code against LangGraph-style state graphs and Devin-style planner-heavy systems.
  2. 5-layer compaction pipeline. Worked example of graduated context management — budget reduction → snip → microcompact → context collapse → auto-compact. Each gated by flags; cheaper layers run first. Most agent papers hand-wave this.
  3. 7-mode permission system + auto-mode classifier + 7 safety layers. Concrete instantiation of deny-first with human escalation, and defense-in-depth with layered mechanisms. Sandboxing reduced prompt frequency by ~84%.
Worth noting up frontScope

The paper is honest that source-level reading can't adjudicate the long-horizon human-capability question. It also doesn't run any experiments — the empirical claims are borrowed from cited work: an RCT study (2025) (16-dev RCT, 19% slower despite perceived 20% improvement); (132-engineer Anthropic survey, 2025); complexity study (2025) (807 Cursor repos, +40.7% complexity); (304K AI-commit study, 2026). Read it as architectural ethnography, not measurement.

The most consequential open question is therefore not how to add more autonomy, but how to preserve human capability over time.

— Section 14, the paper's closing punchline

Context window sizesSection 3.1

200K tokens for older Claude models. 1M tokens for Claude 4.6-series. The window is the binding resource constraint — explaining why the 5-layer compaction pipeline exists.

Four recurring design questions (Section 3.1)

Every harness has to answer these four. The paper's argument is that each answer is a design point, not a neutral implementation detail.

Q1

Where does reasoning live?

Claude Code: the model reasons, the harness executes. Model emits tool_use blocks; query.ts parses, permission-checks, dispatches. A compromised model cannot override sandboxing — the code paths are separated.

Q2

How many execution engines?

Claude Code: one shared queryLoop(). Interactive CLI · Headless · SDK · IDE all converge on it. Only the rendering layer differs. Contrast: SWE-Agent uses mode-specific execution paths.

Q3

Default safety posture?

Claude Code: deny-first with human escalation. "Deny rules override ask rules override allow rules; unrecognized actions escalate to the user rather than allow silently" (permissions.ts). Multiple independent layers evaluate in parallel.

Q4

Binding resource constraint?

Claude Code: the context window. 200K for legacy models, 1M for Claude 4.6-series. Five distinct context-reduction strategies run before every model call — this is why the compaction pipeline exists.

Context as the binding bottleneck (Section 3.6)

Section 3.6Nine context sources compete for a single 200K/1M-token window. Since the window is the binding resource, a five-layer compaction defense runs before every model call. Context management is not a post-hoc optimization — it is the architecture's first problem.

The query pipeline (Section 4.1)

Section 4.1 · the per-turn pipelineEvery call of queryLoop() runs five stages in order: (1) assemble context via memoized getters, (2) shape/compact if pressure, (3) stream the model call, (4) dispatch tools through the permission gate, (5) persist to JSONL and either stop or recurse. This is the pipeline the rest of the mechanisms sit inside.

05Seven-layer permission gate

Deny-first. All applicable layers must pass; any one blocks. Deny is a routing signal.

Section 5.1-5.2 · redrawnSeven-layer permission gate. A tool call must pass all seven layers in order; the first to block denies. Deny is routed back to the model as feedback, not treated as a stop. Layers 4 (ML classifier) and 5 (sandbox) are flagged because they gate the other five.
Section 5.2 · authorization pipelineFour-stage pipeline from the paper's Figure 4 — pre-filter, PreToolUse hook, rule evaluation, permission handler. Stage 4's handler dispatches into one of four branches (Coordinator / Swarm worker / Speculative classifier / Interactive) based on caller context + feature flags; exactly one branch fires per call, producing one of three verdicts (Allow / Ask / Deny) logged to the transcript.

Classifier or deny-rule blocking is treated as a routing signal to the model, not a hard stop — the model receives the denial reason, revises its approach, and tries a safer alternative. Permission shapes behaviour rather than halting it.

rebuilt if MCP config or feature flags change mid-session. assembleToolPool is the single merge point — no other code path splices tools into the request.
Section 6.2 · tools.tsFive-stage merge pipeline: (1) enumerate all ~54 tools; (2) apply mode filter — CLAUDE_CODE_SIMPLE can restrict the surface to just Bash/Read/Edit; (3) strip deny-rule matches before they ever reach the model; (4) merge in MCP tools under mcp__server__tool namespace; (5) on name collision, built-ins win. Tool pool assemblytools.ts

assembleToolPool() is documented as "the single source of truth for combining built-in tools with MCP tools." Five-step pipeline:

  1. Base enumeration — up to 54 tools (19 unconditional + 35 feature-gated)
  2. Mode filtering — CLAUDE_CODE_SIMPLE allows only Bash/Read/Edit
  3. Deny-rule pre-filtering — strips blanket-denied
  4. MCP integration — appState.mcp.tools merged
  5. Deduplication — built-ins take precedence over MCP

Why four mechanisms, not one? (Section 6.3)

MCP, Plugins, Skills, and Hooks are not redundant — they sit at different costs on the context-cost spectrum. Picking the cheapest mechanism that still clears the job is the design discipline.

Section 6.3The four mechanisms are graduated: Hooks pay zero context, Skills pay only a header until triggered, Plugins bundle a team's surface, MCPs pay full-schema for every live connector. The design lesson: match extension weight to how often the capability actually fires. Collapsing them into one mechanism loses this discipline.

CLAUDE.md is user context, NOT system promptCritical design choice

CLAUDE.md content is delivered as a user-role message — compliance is probabilistic. The permission system provides the deterministic enforcement layer. If a rule must be enforced regardless of model judgment, use deny-rules or hooks — not CLAUDE.md.

Auto-memory — 4 types · no embeddingsSection 7.2

Files live in .claude/memory/ as plain markdown with YAML frontmatter. Retrieval is an LLM-based scan of memory-file headers, selecting up to 5 relevant files. No vector DB.

  • user — who the human is (role, preferences, expertise)
  • feedback — rules the user has corrected or confirmed
  • project — state of ongoing work (decays fast)
  • reference — pointers into external systems
Users can inspect auto-memory entries as ordinary files in .claude/memory/, making the agent's learning process transparent and version-controllable.— Section 7.2
Context collapse — view-only projectionSection 4.3 source comment
Nothing is yielded; the collapsed view is a read-time projection over the REPL's full history. Summary messages live in the collapse store, not the REPL array. This is what makes collapses persist across turns.— Section 4.3 source comment

The consequence: durable transcript keeps full fidelity. Future forks / resumes / audits see everything. Only the agent's working view is compressed.

Four non-compaction context-saving decisionsSection 3.6
  1. CLAUDE.md lazy loading — nested directory instructions load only on demand.
  2. Deferred tool schemas — MCP schemas hydrate only when the tool is about to run.
  3. Subagent summary-only returns — deep delegation does not inflate parent context.
  4. Per-tool-result budgets — large outputs truncated at append time.

Why permissions don't restoreSection 9.2
Resume and fork do not restore session-scoped permissions; users must grant them again in the new session. This is a deliberate safety-conservative design choice: sessions are treated as isolated trust domains.— Section 9.2

Restoring previously granted perms would be a convenience win but risks carrying stale trust into a changed context.

Mostly-append · boundary metadataannotateBoundaryWithPreservedSegment()

Compaction never modifies previously written transcript lines. It records headUuid, anchorUuid, tailUuid in the boundary event. Session loader patches the message chain at read time: preserved messages keep original parentUuids on disk; loader uses boundary metadata to link them correctly.

Auditability over query power — every event human-readable, version-controllable, reconstructable without specialized tooling.

10End-to-end walkthrough: “Fix the failing test”

Single trace through sections 3 to 9 to ground how the subsystems compose. Scenario: user asks to fix failing test in auth.test.ts.

  1. Interface. User submits prompt via interactive CLI.
  2. Loop entry. query() → queryLoop(). Memoized getSystemContext() and getUserContext().
  3. Context assembly. Nine layers packed: system prompt + env + CLAUDE.md (+ .claude/rules/*.md) + auto-memory + conversation history.
  4. Pre-model shapers. First turn → no compaction needed.
  5. Model call. Streaming. Model emits tool_use for BashTool({command: "npm test"}).
  6. Permission gate. Layer 1–7 evaluate. BASH_CLASSIFIER on → speculative classifier races timeout. “npm test” low-risk → high-confidence allow. shouldUseSandbox() → sandboxed.
  7. Execute. Sandboxed npm test runs. Output captured; per-tool-result budget caps oversize output.
  8. Subagent delegation. Model delegates via AgentTool({subagent_type: "Explore"}). Isolated context window + Explore's read-only tool set.
  9. Subagent return. Only summary text returns to parent. Full subagent transcript sidechain.
  10. Repair loop. Model reads auth.test.ts, proposes an Edit within acceptEdits-eligible region.
  11. Verification. Re-run npm test. Permission gate re-evaluates each invocation. On success, stop.
  12. Persistence. Every event appended to session JSONL. --resume rebuilds — but permissions don't rebuild.

Three recurring design choices (Section 11.7)

Choice 1

Graduated layering over monolithic mechanisms

Safety = 7 layers. Context = 5-stage compaction + lazy CLAUDE.md + deferred schemas + summary-only subagent return. Extensibility = 4 mechanisms at different costs.

Choice 2

Append-only designs favor auditability over query power

Session transcripts are append-only JSONL with read-time chain patching. Permissions not restored across sessions. Context compaction applies read-time projections, not destructive edits.

Choice 3

Model judgment within a deterministic harness

Trust the model's judgment inside rich deterministic infrastructure. The 1.6% decision-logic ratio captures this quantitatively. Trade-off: good local decisions can produce poor global outcomes under bounded context.

Empirical predictions & early signals (Section 11.4)

Things the paper bets are true but can't prove from source alone. Each is a potential benchmark or measurement.

Pred 1

The 1.6%/98.4% ratio holds as models improve

As frontier capability converges, differentiation will keep migrating out of the model into the surrounding harness. Measurable by re-running the decision-logic accounting per release.

Pred 2

Graduated compaction outperforms RAG for code tasks

Five-layer compaction preserves local structure better than retrieval-only context. Can be tested head-to-head on SWE-Bench-style multi-turn edits.

Pred 3

Deny-first reduces incident rate vs. allow-first

The 84% sandbox-reduction figure is Anthropic-internal. An open RCT across harnesses would substantiate the deny-first posture as a safety mechanism, not just ergonomics.

Pred 4

Subagent isolation preserves reliability at depth

Summary-only returns should flatten error amplification across nested agents. Testable by compositional benchmarks that vary nesting depth while holding task constant.

Emerging architectural directions (Section 11.6)

Architectural decoupling11.6

Treat the harness as an interface independent of any one model family. Swap frontiers without rewriting the deterministic layer. Hooks/skills/MCP already push this direction.

Memory as a first-class subsystem11.6

CLAUDE.md + JSONL is a start, but neither fills the gap between session-scoped and colleague-scale memory. A unit between "transcript" and "wiki" is the open design slot.

Observability & silent failure11.6

Failure modes often go undetected because the agent still produces plausible output. trajectory-anomaly research-anomaly detection and the pass^k metric metric are early moves; observability has to become architectural.

Governance surfaces11.6

EU AI Act takes effect Aug 2026; MIT's AI Agent Index shows only 13.3% of agentic systems publish a safety card. Harnesses will need first-class audit surfaces, not after-the-fact dashboards.

Proactive architectures11.6

Move from reactive "wait for user prompt" to proactive "watch signals and propose action." Requires much tighter approval UX — the user's cognitive budget becomes the binding resource, not tokens.

Long-term capability preservation11.6

The evaluative lens from Section 2.4 returns here as an emerging direction: design the harness so users grow competence over sessions, not atrophy it. Nearly zero existing research instruments this.

Section 13 · Positioning

14Related work & agent taxonomy

Claude Code's design point, mapped against four neighbouring traditions. Minimal scaffolding + layered policy, versus rule-graph orchestration (LangGraph) versus container isolation (SWE-Agent, OpenHands) versus version-control-as-safety (Aider).

Section 13 · positioning mapClaude Code's design point: right of center (model judgment, not rule graphs) and roughly mid-high on scaffolding (layered policy, but no container walls). Contrasts — LangGraph & Devin above (heavier scaffolding, either rule-based or planner-heavy), SWE-Agent & OpenHands to the left (heavier scaffolding via container isolation), Aider bottom-left (lighter, rule-based via git rollback), OpenClaw to the right (different perimeter, gateway trust model).
13.1 Coding agent taxonomy — five dimensionsSection 13.1

The paper taxonomises production coding agents along five dimensions: deployment surface (CLI vs IDE vs server), trust model (per-action vs perimeter), reasoning home (model vs orchestrator), isolation strategy (process vs container vs none), and memory model (append-only vs RAG vs hybrid).

  • Deployment surface — Claude Code: CLI + IDE + SDK. SWE-Agent: container-only. Devin: sandboxed server. Aider: CLI + git.
  • Trust model — Claude Code: per-action deny-first. OpenClaw: gateway DM-pairing. SWE-Agent: container-wall.
  • Reasoning home — Claude Code: in-model + harness dispatch. LangGraph: graph-orchestrated. Devin: planner node.
  • Isolation — Claude Code: shell sandbox + subagent worktrees. SWE-Agent/OpenHands: full container. Aider: git rollback.
  • Memory — Claude Code: CLAUDE.md hierarchy + JSONL. RAG systems: embedding retrieval. OpenClaw: bootstrap files.
13.2 Agent architecture patterns — five threadsSection 13.2
  • Context management — embedding retrieval vs version-controllable files. Claude Code picks the latter.
  • Safety & permissions — SWE-Agent/OpenHands use container isolation; Aider uses git-rollback; Claude Code uses deny-first per-action evaluation.
  • Protocols & extensibility — MCP as extension standard; LangGraph's explicit state-graph orchestration as the alternative.
  • Software architecture — ReAct pattern (ReAct paper (2022)) for agent loops; Anthropic's five composable workflow patterns for higher-level composition.
  • Positioning — "minimal scaffolding with layered policy enforcement" — a distinct point from both rule-based orchestration and container-isolated execution.
Section 14

15Conclusion

Production coding agents can be understood as answers to a recurring set of design questions. Claude Code occupies a clear design point within that space. It gives the model broad local autonomy while surrounding it with a dense deterministic harness for permissioning, tool routing, context compaction, and session recovery.

— the paper (2026), Section 14

The most consequential open question is therefore not how to add more autonomy, but how to preserve human capability over time. Future systems could treat that sustainability gap as a first-class design problem, not a downstream evaluation metric.

— Section 14, closing

17Methodology & 4-tier evidence

Every architectural claim is tagged with an evidence tier. Knowing which tier a claim sits on is how you read this paper responsibly.

Evidence base & four evidence tiers Section 16.1

Section 16.1 · evidence tiersEach claim in the paper is tagged Tier 1 through Tier 4. A Tier 1 claim names an existing function or file; a Tier 4 claim extrapolates from user-research citations. The taxonomy is what lets readers cite responsibly.

Design-space analytic procedure (Section 16.2)

  1. Identify recurring design questions that every production coding agent has to answer.
  2. Map each question to a specific Claude Code implementation choice, via direct source analysis (Tier 1).
  3. Trace each choice back to one of the five stated values and one of the thirteen principles.
  4. Validate against alternatives — how do SWE-Agent, OpenHands, Aider, LangGraph, Devin answer the same question?
  5. Document source locations for every architectural decision so readers can re-verify.
  6. Cross-reference each principle to multiple subsystems for triangulation (a principle that only touches one place is under-instrumented).
  7. Contrast against an independent system (OpenClaw) — this tests whether the design questions generalise beyond one codebase.

Stated limitations of the methodology (Section 16.3)

  • Source-level reading cannot adjudicate runtime behaviour, model-policy interactions, or closed components.
  • Snapshot of v2.1.88 — the system evolves faster than any static analysis.
  • Feature flags create build-time variability that single-version analysis misses.
  • OpenClaw is a single comparator, not an exhaustive survey — findings' generalisation stops there.
  • No access to internal design-decision records or architectural review notes.
Limits

18What the paper does NOT cover

Stating the limits explicitly so I don't over-claim when I cite this.

  • No new benchmarks; no experiments. Empirical claims borrowed from cited work.
  • Source-only analysis. Cannot adjudicate runtime behaviour, model-policy interactions, or closed components.
  • Single-vendor focus. Claude Code v2.1.88 + OpenClaw as the only comparison.
  • No measurement of long-term human-capability claims. Paper acknowledges this is the most consequential open question.
  • No formal verification of safety properties. Paper explicitly names this as an emerging direction.
  • Analytical limitations: memoized getSystemContext() / getUserContext(). Feature flags create build-time variability. bun:bundle tree-shaking forces dynamic require() over static import.

Questions the paper doesn't answer that a researcher could pick up.
